Deel 0

Het grote plaatje

Het spel bestaat uit precies twee bestanden die met elkaar praten over het internet. Onthoud dit ene zinnetje en de helft is al duidelijk:

De server is de scheidsrechter, de browser is het speelveld.
De server (server.js) bewaart de échte spelstand en beslist alles. De browser (index.html) toont mooie 3D-beelden en stuurt jouw acties door.

🧠 server.js — "de baas"

  • Draait op Node.js (buiten de browser).
  • Houdt bij: wie zit in welke kamer, waar staan de schepen, wiens beurt, wie won.
  • Beslist of een schot raak of mis is — de browser mag dat niet zelf bepalen (anders kun je vals spelen).
  • Stuurt updates naar beide spelers tegelijk.

🎨 index.html — "wat je ziet"

  • Draait in de browser van elke speler.
  • Tekent de 3D-oceaan en de schepen met Three.js.
  • Vangt je muisklikken op en stuurt ze als bericht naar de server.
  • Speelt geluiden, toont chat, scorebord en een verborgen debug-venster.

De verbinding: een WebSocket

Een gewone website vraagt een pagina op en is dan klaar. Een spel moet blijven praten: "ik schiet!", "raak!", "nu jouw beurt". Daarvoor gebruiken ze een WebSocket — een open telefoonlijn die in beide richtingen werkt zolang het spel duurt.

Alles wat over die lijn gaat is een klein JSON-pakketje met een type, bijvoorbeeld { "type": "fire", "row": 3, "col": 7 }. De server kijkt naar het type en weet wat te doen.

Bij gewone HTTP moet de browser telkens opnieuw vragen ("is er al nieuws?"). Dat is traag en omslachtig voor een spel. Met een WebSocket kan de server uit zichzelf "de tegenstander heeft geschoten op [3,7]" doorduwen, op het moment dat het gebeurt. Eén keer verbinden, daarna blijft de lijn open.

Deel 2

server.js — blok voor blok

~270 regels, maar het valt uiteen in 7 begrijpelijke stukken. Open telkens de uitleg.

2.1 · Opstarten & gereedschap

server.js
const express = require('express');   // mini-webserver
const WebSocket = require('ws');     // open telefoonlijn
const server = http.createServer(app);
const wss = new WebSocket.Server({ server });
const PORT = 3000;

Hier worden de drie hulpmiddelen ingeladen: express serveert de HTML-bestanden, ws regelt de WebSocket-lijn, en die twee delen samen poort 3000. Onthoud: poort 3000 = het adres waarop het spel luistert.

2.2 · Het geheugen: rooms & connections

server.js
let rooms = {};          // alle spelkamers
let connections = {};    // alle verbonden browsers

function createRoom(id){
  return {
    id, players: [], phase: 'waiting',
    boards: { 0: createEmptyBoard(), 1: createEmptyBoard() },
    shots: { 0: [], 1: [] }, currentTurn: 0, winner: null
  };
}

De server onthoudt alles in twee grote objecten. Een kamer heeft twee spelers (0 en 1), twee speelborden (10×10 raster vol nullen), de lijst met afgevuurde schoten, wiens beurt het is, en een phase. Die fase is een soort stoplicht en doorloopt: waiting → placing → battle → gameover. Bijna elke regel code controleert eerst: "zitten we wel in de juiste fase?"

2.3 · De schepen & de spelregels

server.js
const SHIPS = [
  { name:'Carrier', size:5 }, { name:'Battleship', size:4 },
  { name:'Cruiser', size:3 }, { name:'Submarine', size:3 },
  { name:'Destroyer', size:2 },
]; // 5+4+3+3+2 = 17 vakjes

validatePlacement(board, ships)  // liggen ze legaal? (binnen rand, niet overlappend)
checkSunk(shipCells, shots)       // is dit schip helemaal geraakt?
checkWin(shipCells, shots)        // zijn ALLE schepen geraakt?

Vijf schepen, samen 17 vakjes — daarom zie je in het spel "Treffers: 0 / 17". Je wint zodra je 17 keer raak schiet. De drie functies zijn de scheidsrechtersregels: validatePlacement weigert schepen die buiten het bord vallen of elkaar overlappen, checkSunk kijkt of álle vakjes van één schip geraakt zijn (= gezonken), en checkWin kijkt of de hele vloot weg is.

2.4 · Iemand verbindt

server.js
wss.on('connection', (ws, req) => {
  const cid = ++connectionId;          // uniek nummer per browser
  connections[cid] = { ws, room:null, player:null };
  ws.on('message', raw => handleMessage(cid, JSON.parse(raw), ws));
  ws.on('close', () => { /* speler vertrok → opruimen na 30s */ });
});

Telkens als een browser verbindt, krijgt die een uniek cid (connection-id) en wordt onthouden. Komt er een bericht binnen, dan gaat het naar handleMessage — het kloppende hart. Sluit iemand het tabblad, dan wordt de tegenstander gewaarschuwd en de kamer na 30 seconden opgeruimd als ze leeg blijft.

2.5 · Het kloppende hart: handleMessage

Dit is één grote switch die op het type van het bericht reageert. Elk case is één spelactie:

server.js · handleMessage()
switch (msg.type) {
  case 'joinRoom':  // speler bij kamer voegen (max 2)
  case 'placeShips':// schepen controleren & opslaan
  case 'fire':      // schot: raak/mis bepalen, beurt regelen
  case 'chat':      // berichtje doorsturen
  case 'rematch':   // nog een potje
  case 'getDebug':  // geheime debug-info (sleutel: 'konami')
}
  • joinRoom — Maakt de kamer aan als die nog niet bestaat. Vol (2 spelers)? Dan een error terug. Bij de tweede speler springt de fase naar placing.
  • placeShips — Controleert de plaatsing met validatePlacement. Pas als beide spelers klaar zijn, gaat de fase naar battle.
  • fire — Het belangrijkste. Eerst: is het wel jouw beurt? Al op dat vakje geschoten? Dan kijkt de server op het bord van de tegenstander of er een schip ligt → hit ja/nee → controleert gezonken/gewonnen → en stuurt het resultaat naar iedereen. Bij mis wisselt de beurt.
  • chat — Knipt het bericht af op 200 tekens en stuurt het rond.
  • rematch — Reset de kamer als beide spelers opnieuw willen.
  • getDebug — Geeft interne info terug, máár alleen met de sleutel 'konami'.

2.6 · Naar iedereen tegelijk praten: broadcast

server.js
function broadcast(room, msg){
  const data = JSON.stringify(msg);
  Object.values(connections).forEach(conn => {
    if (conn.room === room.id && conn.ws.readyState === WebSocket.OPEN)
      conn.ws.send(data);            // stuur naar beide spelers
  });
}

Veel berichten (zoals "schot raak op [3,7], nu beurt van speler 1") moeten naar allebei de spelers. broadcast loopt alle open verbindingen af die in dezelfde kamer zitten en stuurt het JSON-pakket. Zo zien beide schermen tegelijk hetzelfde.

💡 In het echte bestand staat hierboven nog een onafgemaakte lus met een // Find by cid...-commentaar — dat is dode code die niets doet; de werkende versie is het stuk hierboven.

2.7 · De deur openzetten

server.js
app.use(express.static(path.join(__dirname, 'public')));
server.listen(3000, '0.0.0.0', () => log('🚢 draait op poort 3000'));

De laatste twee regels zetten het geheel aan: serveer de bestanden uit de map public, en luister op poort 3000 op 0.0.0.0 (= "alle netwerk­adressen", zodat ook een tweede pc in hetzelfde wifi-netwerk kan verbinden). Vanaf nu draait de server.

Deel 3

index.html — blok voor blok

Eén groot bestand met drie lagen: CSS (uiterlijk), HTML (de schermen) en JavaScript (het brein). Het JS is opgebouwd uit drie "klassen" + opstartcode.

3.1 · CSS — het uiterlijk

Bovenaan in <style> staan kleurvariabelen (--ocean-deep, --foam, --gold…) zodat overal dezelfde marine-kleuren gebruikt worden. Daarna styling voor: de schermen (menu, wachten, game-over), de HUD (de hoekpanelen tijdens het spel), de minikaartjes, de chat, de meldingen ("RAAK!"), en het groene debug-venster. Je hoeft de CSS niet regel voor regel te snappen — weet gewoon: dit is puur het uiterlijk.

3.2 · HTML — de schermen die aan/uit gaan

index.html
<div class="screen" id="screen-menu">        // IP + kamer invullen
<div class="screen hidden" id="screen-waiting">  // wachten op speler 2
<div class="screen hidden" id="screen-gameover"> // gewonnen/verloren
<div id="hud"> ... <div id="canvas-container"> // het 3D-beeld

Alle schermen bestaan tegelijk in de pagina; er wordt steeds maar één getoond door de class hidden toe te voegen/weg te halen (functie showScreen()). Daaronder ligt de canvas-container waar Three.js de 3D-wereld in tekent. De HUD-hoeken (speler, kamer, beurt, treffers, besturing) zweven daar bovenop.

3.3 · JavaScript-brein, deel A — AudioEngine

De klasse AudioEngine maakt alle geluiden zelf met de Web Audio API — er zijn geen geluidsbestanden. Een functie note() genereert een toon, noise() maakt geruis. Daarmee worden playExplosion(), playMiss(), playSunk(), enz. samengesteld. Knap: een hele soundtrack uit pure wiskunde.

3.4 · JavaScript-brein, deel B — Scene3D

index.html · class Scene3D
init()        // camera, licht, oceaan, twee 10×10 rasters
updateHover() // welk vakje wijst de muis aan? (raycasting)
handleClick() // klik → schip plaatsen OF schieten
addShipMesh() / addShotMarker() / spawnExplosionParticles()
animate()     // 60x per seconde opnieuw tekenen (golven!)

Scene3D is de hele 3D-wereld. Het bouwt de oceaan en twee borden (jouw kant links, vijand rechts). Met raycasting schiet het een onzichtbare straal vanaf de muis de scène in om te weten welk vakje je aanwijst. handleClick() vertaalt een klik naar een actie afhankelijk van de modus (plaatsen of schieten). animate() draait continu en laat het water golven en explosies opbloeien.

💡 Three.js wordt geladen via <script src="three.min.js">. Dat bestand moet er dus náást staan, anders blijft het scherm zwart.

3.5 · JavaScript-brein, deel C — Game (de regisseur)

index.html · class Game
connect(addr, room)  // open de WebSocket naar ws://IP:3000
send(msg)            // stuur een JSON-pakketje naar de server
handleMessage(msg)   // REAGEER op berichten van de server
fire(row,col) / confirmPlacement() / renderMiniGrids()

Game is de tegenhanger van de server in de browser. connect() opent de lijn: let op de slimme regel const proto = location.protocol === 'https:' ? 'wss' : 'ws' — beveiligde pagina's eisen een beveiligde socket. handleMessage() is een switch die spiegelt wat de server stuurt: joined, phase, fireResult, chat, … en past het scherm aan. Zie je het patroon? Server en client hebben allebei een handleMessage met dezelfde berichttypes — dat is het hele "protocol".

3.6 · De opstartcode & knoppen

Onderaan, in window.addEventListener('load', …), worden alle knoppen gekoppeld: VERBINDEN, ROTEER, BEVESTIGEN, chat versturen, rematch. Ook de sneltoetsen zitten hier: C=chat, Tab=ander camerastandpunt, F1=debug. En een eierei: de Konami-code (↑↑↓↓←→←→ B A) opent automatisch het debug-venster.

Deel 4

Vijf sleutelconcepten

Snap je deze vijf, dan snap je de code. Lees ze, dan zijn de oefeningen makkelijk.

① Client–server & "single source of truth"

De échte spelstand bestaat maar op één plek: de server. De browser is "dom" — hij toont wat de server zegt. Daarom kan een speler niet vals spelen door zijn eigen scherm te hacken.

② Het berichtenprotocol (JSON + type)

Elk bericht is JSON met een type. Server én client hebben een switch (msg.type) die ze allebei begrijpen. Dezelfde namen aan beide kanten = afgesproken taal.

③ State machine: de phase

waiting → placing → battle → gameover. Bijna elke functie checkt eerst de fase. Het voorkomt onzin zoals schieten tijdens het plaatsen.

④ Het rooster als data: 10×10 array

Een bord is gewoon Array(10) × Array(10) vol nullen. 0 = leeg, een scheepsnaam = schip ligt hier. Schieten = kijken of board[row][col] !== 0.

⑤ "Beveiliging" die geen beveiliging is

De debug-sleutel 'konami' staat letterlijk in de code. Iedereen die het bestand opent ziet hem. Het is een leuk eierei, geen echte beveiliging — een goede les over wat wél en níét veilig is.

Deel 6

Dit op GitHub Pages & het spel zelf draaien

✅ Wat WEL op GitHub Pages werkt

Deze leerpagina. Het is puur statische HTML/CSS/JS. Upload het bestand naar je repo en zet GitHub Pages aan (Settings → Pages). Wil je dat dit je homepagina is, hernoem het dan naar index.html. Iedereen kan dan via de link de code leren begrijpen.

⛔ Wat NIET op GitHub Pages werkt

Het spel zelf. GitHub Pages draait alléén bestanden — geen Node.js-programma's. server.js kan er dus niet draaien, en zonder server is er geen WebSocket om naartoe te verbinden. De spelpagina laadt wel, maar "VERBINDEN" mislukt.

▶ Zo speel je het spel wél (lokaal of LAN)

terminal
# 1. Node.js installeren (nodejs.org), daarna:
npm install express ws

# 2. Zet index.html + three.min.js in een map 'public/'
#    en server.js ernaast. Dan:
node server.js

# 3. Open in de browser:
http://localhost:3000
# Tweede speler op zelfde wifi: http://JOUW-IP:3000

Wil je tóch online tegen elkaar spelen, dan heb je een host nodig die Node.js draait (bijv. Render, Railway, Fly.io of een eigen VPS) — niet GitHub Pages.
